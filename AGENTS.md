# Agents

This repository is the home of the **Eidos** standard. `EIDOS.md` is the authoritative definition of the format; everything else supports it.

**Use the skills.** Six live in the top-level [`skills/`](skills) and ship as a Claude plugin: `eidos` (author + validate), `eidos-format` (reshape a rough draft into the spec shape), `eidos-init` (scaffold a new registry), `eidos-property` (add, rename, or retire a custom property), `eidos-domains` (regenerate the `Domains.md` navigation index), and `eidos-migrate` (move specs to a new version). Read the relevant skill (and `EIDOS.md`) before creating, scaffolding, migrating, or reviewing any spec or product doc.

**The form lives in the registry.** A v3 registry owns its shapes and property contract in a hidden `.eidos/` (`shapes/`, `Schema.md`, `Registry.md`). The canonical seed lives, public and front-facing, at the top level in [`standard-seed/`](standard-seed); `eidos-init` installs it into a registry's `.eidos/`, and the other skills read the form from the registry they're working in, not from a copy of their own. There is no top-level `templates/` folder anymore, and a registry's `.eidos/` is committed, never gitignored. A registry also records its naming convention (Title Case, TitleCase, or kebab-case) in `Registry.md`, and the skills locate a registry by its `.eidos/` marker — so the root folder may be named anything (`Blueprint` is just the default).

**Eidos is human-first: facilitate, don't author.** The product owner holds the intent, the scope, and the decisions. Format, supplement, ask clarifying questions, and press on scope; do not generate finished specs or set direction.

**Keep the version in sync.** `EIDOS.md` always holds the current version (its `**Version:**` header and Versioning section). When it changes, update: the badge at the top of `README.md` (e.g. `[Eidos v3.0.0](EIDOS.md)`); the `version` in `.claude-plugin/plugin.json`; the version note in the canonical seed (`standard-seed/Schema.md` and `standard-seed/Registry.md`); and `CHANGELOG.md`.

**On each release (tag),** copy the current `EIDOS.md` into `versions/` under its full semver name (e.g. `versions/v3.0.0.md`) — snapshot at tag time, from the live file, so there's nothing historical to dig up. `EIDOS.md` itself stays the current version. Migrate existing specs with the `eidos-migrate` skill. After changing `EIDOS.md`, `standard-seed/`, `versions/`, or `CHANGELOG.md`, run `scripts/sync-skills.sh` and commit the updated skill copies — they are committed (not gitignored), so a git-marketplace install works on sandboxed hosts like Claude Desktop. `scripts/sync-skills.sh --check` verifies they're current; `scripts/package-plugin.sh` refreshes them before zipping.
